## Problem Statement

Harvester 是一套个人 home lab 信息采集底座。它要长期稳定抓取固定站点，也要支持围绕临时关注主题建立观察面。

第一批已知来源：

- 中国 CDC 健康数据页面：`https://www.chinacdc.cn/jksj/jksj04_14249/`
- 新浪财经 7x24：`https://finance.sina.com.cn/7x24/`
- 临时主题样例：`Unsloth 在 Mac 上微调功能的进展`

系统不是完整智能助手。它负责抓取、存储、去重、索引、可搜索资料库、LightRAG 优先的 KG/RAG 数据准备、失败记录和审计。OpenClaw 等上层 agent 负责搜索、判断、扩展候选来源、写 proposal、调用 Harvester 正式接口。

核心要求：

- 可搜索资料库必须是第一版目标。
- 固定来源和临时主题都要支持。
- 去重必须是第一版目标，包括防止重复抓取和防止重复入库。
- 本地模型优先，模型慢不能阻塞抓取。
- MCP 不作为唯一入口。Agent Skill 是主要 agent 操作界面，OpenClaw 只是首个目标运行环境，MCP 是可选适配层。

## What Makes This Cool

这不是一个“定时 curl 一堆页面”的脚本。它更像个人情报雷达的底座。

你可以说：“最近关注 Unsloth 在 Mac 上微调的进展。”上层 agent 负责找官网、GitHub、Reddit、博客、release、issue 等候选源。Harvester 负责测试这些源，记录来源状态，生成抓取记录，抽取最小内容单元，去重，索引，保留原始证据。后面要推送、日报、重要性判断、LightRAG，都建立在同一份可追溯资料库上。

关键体验：

- 固定站点自己稳定跑，不需要你盯。
- 临时关注主题可以有 TTL，比如 30 天后自动降频或停用。
- 每个 `content_item` 都有来源、抓取时间、原始快照、规范化文本、hash 和索引状态。
- agent 可以提议新增爬虫，但不能绕过审计直接改数据库。

## Constraints

### 已有基础设施

- 一台 mini PC，内网地址大致为 `http://192.168.0.114/`，已部署 Firecrawl，未来可运行 Harvester。
- 当前 Mac Studio，M1 Ultra，128GB 内存，跑本地大模型。
- mini PC上已有 Postgres 和 pgvector。
- embedding 模型使用本地 Qwen 系列模型。
- 另一台 NAS 可用于大对象、归档和备份。

### 设计约束

- 第一版不要引入 Qdrant。Postgres + pgvector 已够用，少一个数据库，系统更容易维护。
- Firecrawl 是 adapter，不是系统事实源。
- Agent Skill 是操作规范，不直接写数据库，也不绑定某一个 agent 宿主。
- 原始抓取结果 append-only 保存，索引、摘要、LightRAG 工作目录和 KG 都可重建。
- 登录站、高反爬站、微博、X、小红书等必须进入受控 recipe 流程，不能让 agent 任意执行脚本。
- 模型调用全部异步化。抓取可以先完成，enrichment 之后补。

## Premises

1. Harvester 是 source of truth。Firecrawl、OpenClaw、本地模型、MCP、其他 agent 宿主都只是外部能力或入口。
2. 原始快照不可变保存。任何摘要、embedding、LightRAG 索引、KG 都是派生数据，可以重算。
3. 固定站点用 `Source Watch`，临时研究方向用 `Topic Watch`。
4. 去重不是优化项，是资料库质量的基础能力，必须在第一版做。
5. 本地模型优先，但必须异步执行，避免模型速度拖垮抓取主链路。
6. Agent 可以提出 source/recipe proposal，但生产抓取配置必须经过 Harvester API 的状态机。
7. 第一版做 B 架构，A 范围：架构按控制平面设计，首批只跑 CDC、Sina 7x24、Unsloth Topic Watch 三个样例。
8. 抓取单元和存储单元必须分开。一个页面、接口响应或截图是 `raw_object`；资料库、去重、搜索、推送的最小单位是从中抽取出的 `content_item`。

## Approaches Considered

### Approach A: 最小可用资料库

用 Postgres 做状态和文本存储，pgvector 做向量检索，Firecrawl 负责普通网页抓取。第一版只支持 CDC、Sina 7x24 和手动 Topic Watch。

Effort: M
Risk: Low
Completeness: 7/10

优点：

- 最快能用。
- 组件少，调试简单。
- 很适合验证个人资料库体验。

缺点：

- 高难站点、登录态、agent 自动写 recipe 会比较粗糙。
- 后续补调度、沙箱、版本化时会返工。

### Approach B: Harvester 控制平面

Harvester 自己管理 source、topic、recipe、crawl run、raw object、content item、item version、dedup、索引任务。Firecrawl、Playwright、本地模型、OpenClaw 等 agent 都是 adapter。

Effort: L
Risk: Medium
Completeness: 9/10

优点：

- 长期结构对。以后加微博、X、小红书、登录态、视觉模型、可迁移 Agent Skill 都顺。
- 每次抓取有审计记录，失败可重试，结果可回放。
- 去重、索引、摘要、LightRAG/KG 都有明确位置。

缺点：

- 第一版表和队列会多一些。
- 需要认真设计 recipe 权限，否则 agent 写脚本会成为安全坑。

### Approach C: Agent 驱动的爬虫工厂

Harvester 暴露工具，上层 agent 负责发现源、生成 recipe、测试抓取、决定是否加入 watch。Harvester 更像执行和存储内核。

Effort: L
Risk: High
Completeness: 8/10

优点：

- 最符合“自己加爬虫、学爬虫”的长期想象。
- 对临时主题很强，比如 Unsloth/Mac 微调这种观察任务。
- 使用体验会很有 agent 感。

缺点：

- 太早做会不稳。上层 agent 会把底层状态搞复杂。
- 没有强审计和沙箱时，自动生成抓取脚本风险高。
- 失败时很难判断问题来自搜索、recipe、执行器、模型还是站点变化。

## Recommended Approach

选择 **Approach B+：Harvester 控制平面 + 可迁移 Agent Skill 作为主操作界面**。

第一版范围按 Approach A 控制，只实现三个样例：

1. CDC 固定周更抓取。
2. 新浪财经 7x24 高频抓取。
3. Unsloth/Mac 微调 Topic Watch。

但数据模型、任务状态和接口按 Approach B 设计。这样不会做成一次性脚本，也不会第一天就陷入完整 agent 自动化。

核心分层：

```text
OpenClaw
  or other agent hosts
  reads Skill, searches web, proposes sources/recipes
        |
        v
Harvester Agent Skill
  workflow rules, safety policy, API/CLI examples
        |
        v
Harvester API / CLI / optional MCP
  source of truth, state machine, audit log
        |
        +--> Firecrawl adapter
        +--> Playwright/browser adapter
        +--> local model adapter on Mac Studio
        +--> Postgres + pgvector
        +--> object storage / NAS archive
```

## Domain Model

### Source

一个可抓取的信息来源。

字段建议：

- `id`
- `url`
- `domain`
- `kind`: `fixed_page | feed | search_result | social_profile | api | topic_seed`
- `status`: `candidate | testing | watched | paused | rejected | archived`
- `trust_level`: `official | high | medium | low | unknown`
- `requires_auth`
- `default_recipe_id`
- `created_by`: `human | agent | harvester`
- `created_reason`
- `last_seen_at`
- `last_success_at`
- `failure_count`

### Topic Watch

一个临时或长期关注方向，不等于 URL。

字段建议：

- `id`
- `title`
- `description`
- `keywords`
- `negative_keywords`
- `seed_queries`
- `status`: `active | cooling_down | paused | archived`
- `ttl_days`
- `refresh_policy`
- `created_by`
- `created_at`
- `expires_at`

例子：

```yaml
title: Unsloth Mac fine-tuning progress
keywords:
  - unsloth
  - mac
  - mlx
  - fine-tuning
  - qlora
negative_keywords:
  - unrelated benchmark spam
seed_queries:
  - unsloth mac fine tuning
  - unsloth mlx support
  - unsloth Apple Silicon training
ttl_days: 30
```

### Raw Object / Fetch Artifact

一次抓取产生的原始证据，不一定等于一篇文档。

例子：

- 新浪 7x24 的瀑布流页面或接口响应。
- X/Twitter 某个用户 timeline 的 HTML、JSON 或截图。
- 微博搜索结果页。
- CDC 列表页。
- CDC 某篇详情文章的 HTML。

字段建议：

- `id`
- `crawl_run_id`
- `source_id`
- `uri`
- `content_type`
- `storage_uri`
- `size_bytes`
- `sha256`
- `captured_at`
- `is_replayable`

`raw_object` 只回答一个问题：这次抓取到底看到了什么。它是证据层，不是资料库的最小内容单元。

### Extraction Run

从 `raw_object` 中抽取一个或多个 `content_item`。

字段建议：

- `id`
- `crawl_run_id`
- `raw_object_id`
- `recipe_id`
- `extractor`: `css | xpath | json_path | readability | llm | vlm | custom_script`
- `status`: `queued | running | succeeded | failed`
- `items_found`
- `items_created`
- `items_updated`
- `items_skipped_duplicate`
- `error_message`

同一个 `raw_object` 可以抽取 0、1 或 N 个 item：

- CDC 列表页可能只抽出详情链接，创建后续抓取任务，`content_item` 为 0。
- CDC 详情页通常抽出 1 篇文章。
- 新浪 7x24 一次页面抓取可能抽出几十条快讯。
- X/Twitter timeline 一次抓取可能抽出几十条 post、reply、quote。

### Content Item

资料库、去重、搜索、推送、LightRAG 的最小内容单元。

字段建议：

- `id`
- `source_id`
- `topic_watch_id`
- `item_type`: `article | news_flash | social_post | comment | release | issue | discussion | attachment | unknown`
- `external_item_id`
- `canonical_url`
- `author`
- `published_at`
- `first_seen_at`
- `last_seen_at`
- `status`: `active | deleted_upstream | hidden | duplicate | archived`
- `parent_item_id`
- `thread_id`
- `first_raw_object_id`
- `latest_raw_object_id`

关键规则：

- 瀑布流页面不是一个 content item。瀑布流里的每条消息才是 content item。
- 时间线页面不是一个 content item。每条 post、reply、quote 才是 content item。
- 列表页默认不是 content item，除非列表页本身就是要保存的公告或报告。
- 一篇长文章、一个 GitHub release、一条微博、一条新浪快讯，都可以统一成 content item。

### Item Observation

一次抓取中观察到某个 `content_item` 的记录。

为什么需要这一层：同一个 raw object 里会有多个 item；同一个 item 又会在多个 raw object 中反复出现。比如新浪 7x24 每分钟抓一次，页面整体 hash 变了，但里面 90% 的快讯是旧的。

字段建议：

- `id`
- `content_item_id`
- `raw_object_id`
- `extraction_run_id`
- `observed_at`
- `position`
- `observed_url`
- `observed_payload_hash`
- `raw_snippet`

规则：

- 新 item 第一次出现时创建 `content_item` 和 `item_observation`。
- 旧 item 再次出现时只创建 `item_observation`，更新 `last_seen_at`。
- 只有正文、标题、附件等规范化内容变化时，才创建新的 `item_version`。
- 搜索和推送默认看 `content_item` / `item_version`，审计和调试看 `item_observation` / `raw_object`。

### Item Version

规范化后的内容版本。

字段建议：

- `id`
- `content_item_id`
- `source_id`
- `topic_watch_id`
- `canonical_url`
- `title`
- `author`
- `published_at`
- `fetched_at`
- `normalized_text`
- `content_hash`
- `simhash`
- `language`
- `raw_object_id`
- `dedup_group_id`

为什么还需要 version：

- 网页会编辑更新，尤其是公告、新闻和文档。
- 社交媒体内容可能删除、改可见性、补充图片或 thread。
- 同一 item 多次抓取时，只有内容变化才创建新 version。
- 搜索默认看最新 version，审计可以回放历史 version。

### Chunk

用于搜索和 embedding 的文本块。

字段建议：

- `id`
- `item_version_id`
- `chunk_index`
- `text`
- `token_count`
- `embedding vector`
- `embedding_model`
- `embedding_status`

### LightRAG-First KG / RAG

优先考虑 LightRAG，而不是 Microsoft GraphRAG 这类更重的 pipeline。LightRAG 当前适合这个项目的原因：

- 它的目标是用图结构增强 RAG，和“围绕 Topic Watch 维护可追溯资料库”的需求一致。
- 它支持 PostgreSQL 相关存储后端，和现有 Postgres + pgvector 方向更贴近。
- 它可以作为异步索引层接入，不需要成为抓取主链路的一部分。

但第一版仍然不要把 Harvester 的事实源交给 LightRAG。Harvester 保存 raw object、content item、item version、chunk、hash、来源和审计日志；LightRAG 是从这些数据重建出来的派生索引。

第一版先存轻量结构，为 LightRAG 做准备：

- `entities`: 人、组织、项目、产品、疾病、政策、股票代码、模型名。
- `entity_mentions`: 某实体在哪篇文档、哪个 chunk 出现。
- `claims`: 从文本抽出的可追溯断言。
- `relations`: `entity_a -> relation -> entity_b`。
- `topic_summaries`: 每个 Topic Watch 的每日/每周摘要。
- `lightrag_index_jobs`: 记录哪些 topic/item batch 已经进入 LightRAG 构建。
- `lightrag_workspaces`: 每个 Topic Watch 或全局语料对应的 LightRAG 工作区元数据。

推荐策略：

- 高频来源，例如新浪 7x24，不要每条快讯实时进 LightRAG。先入库、去重、分批。
- Topic Watch 可以每日或每周构建 LightRAG workspace。
- 官方低频来源，例如 CDC，可以在每次新增文档后触发轻量增量索引。
- LightRAG 失败不能影响抓取和全文检索。失败只生成 `index_failed` job，之后重试。
- 所有 LightRAG 输入必须带 `content_item_id`、`item_version_id` 和 `chunk_id`，回答时能追溯回原文和原始抓取证据。
- LightRAG 的 embedding model 和向量维度必须在 workspace 初始化时固定。更换 embedding 模型时不要原地迁移，创建新 workspace 重建索引。
- 索引阶段尽量不用 reasoning model。索引用稳定、便宜、长上下文足够的本地模型；查询阶段再换更强模型。
- 对中文、英文、代码混合语料，先固定一套 multilingual embedding，再做评估，不要每个 topic 随意换模型。

以后如果要换成 LightRAG 之外的方案，Harvester 的 raw object、content item、item version 和轻量 KG 不变，只换索引 adapter。

## Dedup Strategy

去重分四层：抓取调度去重、raw object 去重、抽取时 item 身份去重、入库后内容/语义去重。

### 1. 抓取调度去重：防止重复抓取

目的：不浪费抓取资源，不让高频站点刷爆。

机制：

- `source_schedule_lock`: 同一 source 同一时间窗口只能有一个 run。
- `fetch_fingerprint`: 对 `canonical_url + recipe_version + normalized_params + time_bucket` 做 hash。
- `If-None-Match` / `If-Modified-Since`: 如果站点支持，优先使用。
- `min_interval_seconds`: 每个 source 设置最小抓取间隔。
- 高频页面使用“列表页发现 + 详情页增量”的策略。

新浪 7x24 特殊处理：

- 列表页可以高频抓。
- 每条新闻按标题、时间、正文短 hash 去重。
- 若站点接口可稳定解析，优先抓接口而不是渲染页面。
- 只把新 item 入库，不要每次把整页当一篇新文档。

### 2. Raw Object 去重：判断这次抓取证据是否重复

目的：判断页面、接口响应、截图这类抓取证据是否和上次完全一样或近似一样。

机制：

- `raw_object.sha256`: 原始响应 bytes 的强 hash，完全相同则可标记为 duplicate raw object。
- `normalized_raw_hash`: 去掉时间戳、广告、随机 nonce、空白后的页面级 hash。
- `etag` / `last_modified`: 站点提供时保存，用于后续 conditional request。
- `raw_object_dedup_group`: 多个 raw object 完全相同或近似相同时归组。

关键点：

- raw object 重复，不代表 item 一定重复，但通常可以跳过抽取。
- raw object 变化，不代表里面都是新 item。瀑布流和时间线经常只是新增几条、位置变化、广告变化。
- 高频源不要因为 raw object 变化就把整页写成新资料。必须进入 item extraction。

### 3. 抽取时：防止重复 item

目的：同一次瀑布流、时间线、列表页重复出现旧消息时，不重复创建 item。

机制：

- `external_item_id`: 如果站点有稳定 ID，优先用它，例如微博 ID、tweet ID、GitHub issue ID。
- `source_item_key`: 对 `source_id + external_item_id` 建唯一约束。
- 没有稳定 ID 时，用 `canonical_url + published_at + normalized_title/body prefix` 生成弱 key。
- 同一 `raw_object` 抽出的 item 先在内存中去重，再写入数据库。
- 已存在 item 不创建新 item，只写 `item_observation`，更新 `last_seen_at`。
- 内容变化才创建新 `item_version`。
- 如果页面有游标、since_id、max_id、高水位时间戳，可以用于减少扫描，但不能只依赖游标。社交站和新闻流会重排、置顶、回填、删除。

这层解决新浪 7x24、X/Twitter timeline、微博列表、小红书搜索结果这种场景。

新浪 7x24 的典型流程：

1. 每分钟抓一次列表页或接口，生成一个 `raw_object`。
2. 即使 raw object hash 变了，也只是说明页面证据变了。
3. extractor 从 raw object 中抽出 N 条快讯。
4. 每条快讯按稳定 ID 或弱 key upsert `content_item`。
5. 旧快讯只新增 `item_observation`，不新增 item，不新增 version。
6. 新快讯创建 `content_item` 和第一版 `item_version`。
7. 如果旧快讯正文被修正，创建新的 `item_version`。

### 4. 入库后：防止重复内容污染资料库

目的：同一内容出现在多个站点、多次抓取、多种 URL 时，不污染搜索和摘要。

机制：

- `canonical_url_hash`: URL 规范化后 hash。
- `content_hash`: item version normalized text 的强 hash，完全重复直接合并。
- `simhash`: 近重复检测，用于标题变体、转载、参数页。
- `dedup_group`: 多个 item version 指向同一内容组。
- `source_priority`: 官方源优先，转载源作为补充证据。

去重结果不删除原始记录。只是在搜索、摘要和 Topic Watch 里默认按 dedup group 展示。

## Storage Design

## First Three Example Flows

### CDC Weekly

目标：每周抓取 CDC 健康数据页面，保留历史变化。

策略：

- 固定 `Source Watch`。
- 每周运行。
- 抓取列表页，识别新增文章链接。
- 对新增详情页单独抓取。
- 列表页保存为 `raw_object`，通常不进入资料库搜索。
- 详情页抽取成 `content_item(item_type=article)`。
- 文章 item 按标题、发布时间、正文 hash 去重。
- 官方源 `trust_level=official`。

成功标准：

- 每周能发现新增记录。
- 同一文章不会重复出现在搜索结果里。
- 原始 HTML 和规范化正文都可回放。

### Sina 7x24

目标：高频抓取财经快讯，建立增量时间线。

策略：

- 不把整页当文档。整页或接口响应只是 `raw_object`。
- 每条快讯是一个 `content_item(item_type=news_flash)`。
- 抓取频率可以 1-5 分钟起步，后续按需要调整。
- 去重 key：`published_at + normalized_title + normalized_body_hash`。
- 如果能找到稳定接口，优先接口抓取；否则 Firecrawl 或 Playwright。
- 已见过的快讯只更新 `last_seen_at`，内容变化才创建新 item version。

成功标准：

- 高频抓取不产生大量重复 item。
- 能按时间范围搜索。
- 抓取失败不会影响已入库快讯。

### Timeline / Waterfall Sources

目标：统一处理新浪 24 小时、X/Twitter timeline、微博主页/搜索、小红书搜索结果等“一个页面下面很多消息”的来源。

策略：

- Source 代表入口页面或接口。
- Crawl Run 代表一次抓取。
- Raw Object 保存整页 HTML、接口 JSON、截图等证据。
- Extraction Run 从 Raw Object 里抽出 N 个 Content Item。
- Item Observation 记录“这次 raw object 中又看到了这个 item”。
- Content Item 才参与去重、搜索、推送和 LightRAG。
- 如果站点有稳定 item ID，必须保存 `external_item_id`。
- 如果没有稳定 ID，用弱 key + simhash，降低重复污染。

成功标准：

- 同一个 timeline 多次抓取不会重复创建旧消息。
- raw object 整体变化时，旧 item 只新增 observation，不新增 item。
- 删除或不可见的上游消息不会让本地历史丢失，只更新状态。
- 搜索结果展示单条消息，不展示“整页瀑布流”。

### Unsloth Mac Fine-Tuning Topic Watch

目标：临时关注 Unsloth 在 Mac/MLX/Apple Silicon 上微调支持的进展。

策略：

- 创建 30 天 TTL 的 Topic Watch。
- 候选源包括官网、GitHub repo、GitHub releases、issues/discussions、Reddit、相关博客。
- 候选源先进入 `candidate`。
- 上层 agent 可以提出来源，但 Harvester 测试抓取质量。
- 每日或每周生成 topic summary。

成功标准：

- 能看到“最近新增了哪些实质进展”。
- 能追溯到原文来源。
- 30 天后自动降频或暂停。

### Accepted Decisions

1. **API safety:** 第一版只做 API token 和 audit，不做完整 fetch policy。fetch policy 已写入 `TODOS.md`，排在 LightRAG/KG/MCP 后。
2. **Topic model:** `topic_watches` 和 `topic_sources` 进入 MVP，但不做自动 discovery。
3. **Job queue:** Postgres job table 必须包含 lease、retry、`idempotency_key`、dead 状态，并用 `FOR UPDATE SKIP LOCKED` claim。
4. **Deployment:** 第一版必须有 Compose 配置校验、migration 测试、server/worker healthcheck 和 smoke。
5. **State machine:** source/topic/recipe/run 状态转移集中处理，所有状态变更写 audit。
6. **CLI:** Typer CLI 通过 HTTP API 操作，不直接写数据库。
7. **Tests:** 采用完整 MVP 测试层：migration、API 集成、队列并发/lease、CDC/Sina fixture、CLI smoke、Compose smoke。
8. **Embedding:** `raw_object` 不做 embedding。只对抽取、规范化、去重后的 `item_version -> chunks` 做 embedding。
9. **Search:** 第一版定义索引契约：latest item version、source/topic/time 过滤、关键词 full-text 或 trigram、pgvector、默认 dedup collapse。
10. **Frontier:** 高频源加入 `source_frontiers` 或等价字段，记录 `cursor`、`frontier`、`rewind_window`、`last_complete_range`，按“前进 + 重叠回扫 + 去重”执行。
11. **Pipeline consistency:** 每个 pipeline stage 定义事务边界、唯一 upsert key、下游 job outbox 或同事务创建规则。
12. **Raw retention:** raw HTML/API payload 默认短保留，例如 7 天。提取成功且 item/version/chunk 已落库后，可以压缩或删除 payload，长期保留 metadata、hash、审计和必要 URI。关键来源可单独提高保留级别。
13. **Queue fairness:** job runner 加 `per_source max_in_flight`、`per_job_type concurrency`、`queue depth cap`，人工 test 和失败重跑有保护配额或独立 claim lane。
14. **Stable tests:** CDC/Sina 回归用固定 `raw_object` fixture、冻结时间、stubbed Firecrawl/model output。真实 Firecrawl/Qwen 只做少量 smoke。
15. **URL normalization:** schema 区分 `original_url`、`final_url`、`canonical_url`、`canonical_url_hash`，并测试 tracking 参数、fragment、redirect 和移动站变体。

### Performance Requirements

- `raw_object` payload is short-retention evidence cache, not embedding input.
- Sina/high-frequency sources use batched observation writes and dedup before enrichment.
- Embedding jobs are throttled by job type and queue depth.
- Job runner enforces per-source and per-job-type concurrency.
- Search indexes are part of MVP migration, not a later migration after data growth.
- High-write tables must have retention policy and migration structure ready for time partitioning or rolling cleanup.

Parallel lanes:

- Lane A: DB base -> API/state machine -> CLI.

Execution order:

1. Build DB base first.

Conflict flags:
